# Agents.md — Tsumiki Multi-Agent Specification

This document defines the agent architecture in technical detail: what each agent does, what state it reads/writes, how it makes decisions, and how agents hand off to one another. The system is implemented as an explicit **state graph (LangGraph)** — every agent is a node operating on a shared, typed state object, not a chat persona.

---

## 0. Shared State Object

All agents read from and write to a single typed state object (passed through the graph). This is what makes hand-offs inspectable and demoable — you can print/log this object at every node transition.

```python
class UserState(TypedDict):
    user_id: str
    active_goal: Goal                # current goal: title, domain, target_date, milestones[]
    plan: list[PlannedAction]        # current week/day plan generated by Planner
    checkin_history: list[CheckIn]   # timestamped logs of completed/missed actions
    streak: int
    last_checkin_at: datetime
    reflection_notes: list[str]      # qualitative insights surfaced by Reflection Agent
    world_state: WorldState          # stack height, stone variants, "balance" status
    pending_event: AgentEvent | None # structured event waiting to be applied by the Game Engine
    escalation_level: int            # 0 = none, 1 = notification, 2 = voice call
```

Two memory backends back this object:
- **Relational store** (SQLite/Postgres): `Goal`, `Milestone`, `CheckIn`, `WorldState` — anything structured and queryable.
- **Vector store** (Chroma): free-text reflections and qualitative notes, retrieved by semantic similarity (e.g., "user struggles with mornings").

Agents query only the slice of memory relevant to their job — this is what keeps responses specific and grounded instead of vague.

---

## 1. Planner Agent

**Purpose:** Converts a high-level goal into a concrete, time-boxed plan of milestones and daily/weekly actions.

**Reads:** `active_goal`, `reflection_notes` (if any exist from prior cycles), relevant goal-domain knowledge (via retrieval if needed).

**Writes:** `plan` (list of `PlannedAction` objects, each with a date, description, estimated effort, and linked milestone).

**Decision logic:**
1. Decompose the stated goal into milestones with realistic target dates.
2. Break the nearest milestone into specific daily/weekly actions.
3. If `reflection_notes` exist (e.g., "user consistently skips Monday sessions"), adjust the plan to route around known friction points *before* they recur — this is what makes the system feel like it actually knows the user.

**Hands off to:** Accountability Agent (which will track adherence to this plan going forward).

**Example trigger:** New goal created, or a milestone is completed and the next phase needs planning, or the Reflection Agent flags that the current plan isn't working.

---

## 2. Accountability Agent

**Purpose:** The system's "eyes on the ground." Monitors check-ins against the plan and decides how — and whether — to intervene.

**Reads:** `plan`, `checkin_history`, `streak`, `last_checkin_at`, `escalation_level`.

**Writes:** `escalation_level`, generates intervention content (notification text or voice-call script), and may emit a `pending_event` (e.g., `setback_detected`, `streak_maintained`) for the Game Master Agent.

**Decision logic (escalation ladder):**
| Condition | Action |
|---|---|
| Check-in completed on schedule | Emit `streak_maintained` event; no intervention needed |
| One missed check-in | Send a warm, specific notification referencing the actual plan item ("Looks like today's 20-min Spanish session got skipped — want to do a shorter version tonight?") |
| Two consecutive missed check-ins | Increase `escalation_level` to 1; notification tone shifts to gentle check-in ("Everything okay? Your last session was 2 days ago.") |
| Three+ consecutive missed check-ins, notifications unanswered | Increase `escalation_level` to 2; **autonomously decide to place a voice call** via the Voice Escalation flow (see §5), generating a personalized script from `checkin_history` and `active_goal` |
| User resumes check-ins after a gap | Reset `escalation_level` to 0; emit `comeback_detected` event (acknowledged warmly, not punitively — no "you broke your streak" messaging) |

**Important design choice:** the Accountability Agent never frames missed check-ins as failures. Its language always treats a gap as information to plan around, not a moral lapse — this is core to the "calm, not guilt-driven" positioning.

**Hands off to:** Reflection Agent (logs the pattern for future analysis), Game Master Agent (structured event), and — at escalation level 2 — the Voice layer.

---

## 3. Reflection Agent

**Purpose:** The system's long-term memory and pattern-recognition layer. Runs periodically (e.g., weekly) rather than per-interaction.

**Reads:** `checkin_history` (relational), past qualitative notes (vector store query for semantically similar past patterns).

**Writes:** `reflection_notes` (new qualitative insights, stored in both the relational summary table and the vector store for future semantic retrieval).

**Decision logic:**
1. Analyze check-in timing/frequency for patterns (e.g., "80% of missed sessions are on Mondays," "user performs best in morning slots").
2. Cross-reference with any qualitative notes the user has logged ("felt drained after work").
3. Produce concise, structured insights and store them — tagged by goal domain — so the Planner can use them on the *next* planning cycle.

**Why this matters:** this is the agent that makes Tsumiki feel like it "remembers you" rather than starting fresh each session — directly answering the "ChatGPT forgets everything" complaint that motivated this whole project.

**Hands off to:** Planner Agent (on the next planning cycle) and, in the Shared Circles feature, can surface gentle social prompts (see §6).

---

## 4. Game Master Agent

**Purpose:** Translates agent decisions into the visual world — but **does not control visuals directly**. It only emits structured events; a separate deterministic engine (plain code, not an LLM) interprets them.

**Reads:** `pending_event` (set by Accountability Agent or milestone-completion logic).

**Writes:** Confirms and enriches the event payload (e.g., attaches a stone "variant" tag based on the difficulty/type of the achievement) and passes it to the **Tsumiki Engine**.

**Why this agent exists as a separate node (rather than just letting the Accountability Agent emit events directly):** it creates a clean boundary between "agents that exercise judgment" (planning, tone, escalation decisions) and "code that guarantees deterministic, demoable outcomes" (the visual). This separation is a deliberate engineering decision, and a good one to call out explicitly to judges — it shows you know which parts of a system should be probabilistic and which must be reliable.

```python
class AgentEvent(TypedDict):
    type: Literal["milestone_reached", "streak_maintained", "setback_recovered", "comeback_detected", "support_received"]
    goal_id: str
    difficulty: Literal["small", "medium", "large"]
    timestamp: datetime
```

**Hands off to:** the **Tsumiki Engine** (deterministic, outside the agent graph) — see below.

## 5. Voice Escalation Flow (triggered by Accountability Agent at escalation_level = 2)

This is not an "agent" in the LangGraph sense — it's an external action the Accountability Agent autonomously decides to take.

1. Accountability Agent generates a short, personalized call script from `active_goal` + `checkin_history` (e.g., referencing the user's actual streak and goal).
2. The script is passed to **Vapi**, which handles telephony (via Twilio under the hood), real-time speech-to-text, and text-to-speech — Tsumiki's own agents remain the "brain"; Vapi is the "voice."
3. Call outcome (answered, voicemail, user response) is logged back into `checkin_history`/`reflection_notes`, closing the loop — the system learns from how the user responds to escalation, too.

**Demo safeguard:** pre-test the flow extensively; keep a recorded backup clip in case live telephony has issues during judging.

## 6. Shared Circles (social layer — extends Reflection + Accountability, no new agent needed)

Rather than adding a dedicated "social agent," the social layer is a thin extension of existing agents:
- The **Reflection Agent**, during its periodic run, can notice that a member of the user's circle has gone quiet and gently surface it: *"Looks like Asha hasn't checked in for two days — want to send her a stone of support?"*
- A "stone of support" is a lightweight, wordless social action: one user nudges another, and the recipient's `world_state` receives a small, clearly-marked addition (visually distinct from self-earned stones).
- No leaderboards, scores, or comparisons — circles are about presence and quiet encouragement, consistent with the calm/non-competitive positioning.

This keeps the social feature essentially free from an architecture standpoint: same agents, same state object, one new event type (`support_received`), and a small UI addition (seeing others' stacks + a "send support" button).

## 7. The Tsumiki Engine (deterministic — outside the agent graph)

Plain rules-based code (no LLM) that consumes `AgentEvent` objects and updates `world_state`:

```python
def apply_event(world_state: WorldState, event: AgentEvent) -> WorldState:
    if event["type"] == "milestone_reached":
        world_state.add_stone(variant=stone_variant_for(event["difficulty"]))
    elif event["type"] == "streak_maintained":
        world_state.reinforce_balance()
    elif event["type"] == "setback_recovered":
        world_state.add_stone(variant="resilient")   # marks comebacks positively, not punitively
    elif event["type"] == "support_received":
        world_state.add_stone(variant="gifted", from_user=event.get("from_user_id"))
    return world_state
```

This function is unit-testable, has no external dependencies, and behaves identically every time — exactly what you want running live in front of judges.

## 8. End-to-End Flow (what the demo actually shows)

```
User sets a goal
   → Planner Agent generates plan (reads reflection_notes if any)
   → [days pass; user checks in or doesn't]
   → Accountability Agent evaluates adherence
       → emits AgentEvent  → Game Master Agent enriches it → Tsumiki Engine updates world_state (visible change)
       → if escalation needed → generates call script → Vapi places call → outcome logged
   → Reflection Agent (periodic) analyzes patterns → writes reflection_notes → feeds back into next Planner cycle
   → (optional) Shared Circles: Reflection Agent surfaces a social nudge → "support" event → Tsumiki Engine updates a friend's stack
```

Every arrow in this diagram corresponds to an inspectable state change you can show live — which is the core of the "we are not a wrapper" argument.

## 9. Tech Notes

- **Framework:** LangGraph for the agent graph; each agent is a node with a clearly defined input/output schema (above).
- **Models:** Claude/GPT via API — keep provider-agnostic where possible (LangGraph supports swapping models per node).
- **Memory:** SQLite/Postgres (relational) + Chroma (vector) — both queried with narrow, typed requests, never "dump the whole history into the prompt."
- **Engine:** plain Python, fully unit-tested, zero LLM calls.
- **Voice:** Vapi (wraps Twilio + STT/TTS) — Tsumiki supplies the content, Vapi supplies the channel.
